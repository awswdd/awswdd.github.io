工程笔记

把不可信 agent 放进 Firecracker：零凭证 microVM 的工程账

内容创作：wdd · 美化编辑：codex (gpt-5.5) · 审核：wdd

约 3655 字 · 阅读约 10 分钟

我构建的是一个持续会话式的 agent 运行环境。每个用户有一个 workspace，agent 可以读写文件、调用工具、安装依赖、执行命令；在 coding 场景下，它还会运行 bash、npm install、构建脚本和测试脚本。

这个模型里，隔离边界不能只停在进程和容器。Docker Engine security 讲得很清楚：容器隔离依赖 namespaces、cgroups、capabilities、mounts 等 Linux 机制。对普通业务服务，这是很好的工程权衡；对“模型驱动的未知操作链 + 用户 workspace + 多租户”，我不把共享宿主内核当最后一道墙。

Firecracker 在这里解决两件事：启动快，隔离硬。但真正花时间的不是“把 VM 拉起来”。真正的工程账在后面：文件怎么进出、网络怎么收口、凭证为什么不能进 VM、agent 怎么分发、snapshot 恢复后怎么挂盘、workspace 又怎么持久化。

Firecracker 为什么又回到牌桌上

Firecracker 不是新技术。AWS 在 2018 年发布并开源 Firecracker，目标就是给 Lambda、Fargate 这类多租户 serverless 平台同时提供 VM 级隔离和低冷启动成本。同一时期，Kata Containers 也在把轻量 VM 做成“像容器一样使用”的安全容器运行时。

它后来没有替代普通容器，原因很直接：大多数服务不值得为 VM 隔离额外承担 rootfs、kernel、块设备、网络模型、snapshot 和 host 侧代理的复杂度。容器在工程上太顺手，生态也太完整；只要 workload 是你自己构建、自己部署、自己运维的服务，容器就是更便宜的默认选项。

agent sandbox 把这个老问题翻了出来，不是因为大家突然重新发现了 Firecracker，而是 workload 变了。平台能控制 VM 模板、设备模型和出向代理，VM 内运行的却是模型驱动的未知命令序列。这个形态更接近 provider-controlled runtime 里的不可信 workload，而不是普通服务部署。

这个判断也能从外部生态看到。Docker 的 AI Sandboxes 文档直接写到 “AI coding agents in isolated microVM sandboxes”；它的安全文档把 primary trust boundary 放在 microVM 上，并明确说 raw credentials never enter the VM，网络经 host 侧代理出去。

Firecracker 的取舍也足够明确。Firecracker specification 写到，在最小内核和 rootfs 下，从 InstanceStart 到 guest /sbin/init 可以做到 125ms 以内，VMM 线程内存开销不超过 5MiB。Firecracker design 把 guest vCPU 线程视为启动后即可运行恶意代码，隔离依赖 KVM、seccomp、cgroups、namespaces 和 jailer。FAQ 列出的设备很少：virtio-net、virtio-balloon、virtio-block、virtio-vsock、serial console 和一个最小 keyboard。少设备降低攻击面，也会在后面直接限制共享目录、热插拔和网络形态。

我在裸金属机器上做过一组项目内实测。这不是 Firecracker 官方承诺，也不是通用 benchmark，只用于判断 VM 能不能进入在线路径：

指标 实测结果
VM 冷启动 1.31s
snapshot resume 0.22s
guest 内 ext4 经 virtio-blk 写 1GB 1.2 GB/s
guest 内 ext4 经 virtio-blk 读 1GB 3.1 GB/s
npm install 约 880 个包 22s

后续预热池把 Firecracker VM “取用到 ready”的时间做到约 443ms。这个数字让架构边界可以收紧：读文件、写文件、grep、执行命令、跑一个 agent turn，都先确保 VM 在，再在 VM 内完成。对象存储退出在线读写路径，只保留为持久化真源。

这一步很关键。在线路径如果绕过 VM，系统会有两套文件语义：对象存储里一份，VM ext4 里一份；控制面能写，VM 也能写。后面的同步、冲突和写权归属会越来越复杂。收进 VM 后，/workspace 的 ext4 就是当前态，VM 是唯一在线写者，host 只负责调度、代理和持久化。

VM 只负责执行，不负责持有能力

我把运行时边界定成一句话：VM 是执行边界，不是能力边界。

agent 在 VM 里读写 /workspace、执行命令；云凭证、对象存储权限、模型服务真实凭证、网络出口策略留在 host 侧。VM 被攻破时，攻击者只能拿到 VM 内可见的工作区状态，拿不到云账号能力。

第一步是去掉普通网络面。早期实现里还有 TAP、veth、network namespace、NAT、iptables；后来全部砍掉。现在 VM 没有 eth0、没有 NAT、没有公网路由。管理通道和出向请求都走 vsock。Firecracker 的 vsock 文档定义了 guest AF_VSOCK 到 host Unix domain socket 的通信模型；design 文档也明确说 Firecracker 不做网络过滤，guest 出站流量应由 host 过滤。我的选择是再往前走一步：不给 guest 配普通网卡，让所有出向都先变成一条 host 可识别、可审计、可拒绝的 vsock 请求。

VM 内的 agent 仍然使用普通 HTTP 客户端，请求发到 127.0.0.1:8090。这个 loopback 端口由 guest-agent egress shim 监听，再桥到 host 的 vsock 端口：

VM 内 agent
  -> 127.0.0.1:8090
  -> guest-agent egress shim
  -> AF_VSOCK
  -> host egress proxy
  -> 被允许的 upstream

guest 侧没有路由选择权，host 侧 egress proxy 决定放行、身份和重签。

第二步是凭证不进 VM。VM 里没有 /root/.aws，没有 AWS 访问密钥环境变量，agent 卷不包含 .pem 或 secret；临时凭证也不进 VM，因为在“不可信 agent 可执行任意命令”的前提下，临时凭证只是有效期更短的可窃取能力。

模型调用路径压成四步：

  1. VM 内 agent 发请求，请求里只有占位凭证，例如 AWS_ACCESS_KEY_ID=agent-egress-placeholder。
  2. 请求经 loopback shim 和 vsock 到 host egress proxy。
  3. egress proxy 用 host 侧真实凭证重新签名，再访问 Bedrock 等上游服务。
  4. VM 自始至终摸不到真实凭证。

占位值只用于兼容要求环境变量存在的 SDK，不参与认证。

重签还有一个容易踩坑的细节：AWS SigV4 文档要求签名覆盖 canonical request，并提醒不要把 connection、user-agent、transfer-encoding、upgrade 这类 hop-by-hop header 放进签名。Bedrock 路由会丢弃 VM 请求里的 Authorization 和旧 x-amz-*，只保留受控 content-type，再由 host 签名器生成新的 authorization / x-amz-*。否则中间层或 HTTP 客户端改一个 header，就会触发 SignatureDoesNotMatch。

身份也不靠请求体自报。每个 VM 在 jailer chroot 里有自己的 vsock UDS 和一个 sidecar identity 文件，内容是 tenant_id、workspace_id、vm_id。egress proxy 周期扫描这些 sidecar，为每个 VM 起一条专属 UDS listener。请求从哪条 UDS 进来，身份就固定在哪个 VM 上。keep-alive 多请求、恶意请求头、请求体自报身份，都不能改变连接级身份。

最终数据流只有三条：

文件读写：VM 内 agent -> /workspace ext4
模型调用：VM 内 agent -> vsock -> host egress proxy -> 上游模型服务
持久化：host -> dm-snapshot -> aws s3 sync -> 对象存储

只有第一条发生在 VM 内；后两条都在 host 边界上完成。

agent 进 VM，但不进 rootfs

当 workspace 的读、写、执行都已经经 VM，agent loop 继续留在 host 会制造一层多余的工具网关。早期 loop 在 host，通过 write_file、apply_patch、grep 这类受限工具把操作送进 VM。这个设计在“host 侧 loop 才是可信执行者”的阶段是合理的；但当文件、命令、项目上下文都进入 VM 后，这层网关就只剩维护成本。

我把 agent loop 搬进 VM。安全模型随之改变：不再试图证明每个工具 API 都拦得住，而是让 agent 在 VM 内直接 open 文件、执行命令；外层用 KVM 边界、零网络面、凭证不进 VM、出向 broker 来收住能力。

接下来要解决的是 agent 怎么进入 VM。

把 agent 烤进 rootfs 最省事，也最不适合长期运行。每次 agent 升级都要重烤镜像；多个 agent、多个版本一多，rootfs 会越来越胖，发布和回滚也会变慢。项目里曾经把某个 agent CLI 和 Node runtime 烤进 rootfs，镜像约 1.7GB。拆成 agent 卷后，rootfs 降到约 785MB，只保留 OS、init、sshd 和 guest-agent。

rootfs 和 agent 卷的职责要分开：

  • rootfs 负责 VM 怎么启动、怎么挂 workspace、怎么和 host 通信，应该稳定、少变、少重烤。
  • agent 卷负责这个 workspace 运行哪个 agent、哪个版本、带哪些 runtime，应该能快速发布、回滚和复用。

Firecracker 也把分发方式限定得很清楚。design 文档的 storage 部分说 block devices 由 host 文件作为 backing；README也把磁盘描述为 file-backed block device。相反，把 host 文件系统共享进 guest 仍然是单独的开放议题，可以看这个 Host Filesystem Sharing issue。生产稳定路径是 virtio-blk，不是共享目录。

最后的形态是：每个 agent × 版本 打成一个只读 ext4 镜像，经 virtio-blk 挂进 VM，例如 /opt/agents/0、/opt/agents/1。workspace 也是 ext4，经 virtio-blk 暴露；区别是 workspace 可写，agent 卷只读。

发布链路因此变成：

agent 发布 -> 生成只读 ext4 镜像 -> host cache/content hash
VM 激活 -> 选择 agent 镜像 -> 挂到预留只读槽
guest-agent -> mount /opt/agents/N
agent turn -> 注入 PATH 或可执行文件路径

只读不是装饰。agent 是平台发布物，应该可复现、可缓存、可回滚；workspace 才是用户可写状态。多个 VM 指向同一个只读 backing file，还能复用 host page cache。新版本使用新内容路径，不需要在运行中的 VM 里做缓存失效。

不能热加新盘，就预留占位槽

agent 卷确定走 virtio-blk 后，下一个假设是：从 golden snapshot 恢复 VM，需要哪个 agent，就临时给它加哪块盘。这个假设在实测里不成立。

Firecracker v1.16 的 MMIO 路径上，对 snapshot 里从未声明过的新盘执行：

PUT /drives/vdNEW   -> 400 PCI is not enabled
PATCH /drives/vdNEW -> 400 Device not found

也就是说，在我使用的 MMIO 快照路径里，运行中或 resume 后不能新增一块全新的 virtio-blk 盘。这不等于 Firecracker 完全没有 hotplug。device hotplug 文档写得很清楚：hotplug 目前是 Developer Preview，要求启动时启用 --enable-pci，并且 “Device hotplugging is not supported with MMIO transport”。官方 PCI hotplug 的存在，不改变 MMIO snapshot 路径的约束。

设备数也有限。当时的 MMIO 配置下，第 17 个块设备启动时报：

MMIO transport error: Failed to allocate requested resource

所以块设备不能当成无限插件系统。它适合挂 workspace、少量 agent 卷，不适合承载“10 个 agent × 100 种软件 × 多版本”的任意组合。

最终方案是把动态需求前移到模板期：生成 golden VM 时，预先声明 workspace 槽和若干 agent 占位槽；拍 snapshot 时，这些设备已经被 Firecracker 认识，但 guest 还没有把它们挂载成真实目录。激活 workspace 时，从 golden LoadSnapshot，microVM 进入 Paused 状态；host 用 PATCH /drives/{id} 把占位 backing file 换成真实 workspace ext4 和 agent ext4，再 resume。

这个时机来自 Firecracker 自身的行为。snapshot 文档写到，LoadSnapshot 成功后 microVM 处于 Paused 状态，需要 resume 才会运行；同一份文档还说明 snapshot 里的 disk files 由使用者管理。README也把“change the backing file for a block device, before or after the guest boots”列为能力之一。

一次激活的顺序是：

host:  LoadSnapshot(golden)  -> microVM Paused
host:  PATCH workspace slot  -> 真实 workspace ext4
host:  PATCH agent slot 0    -> 主 agent 只读 ext4
host:  PATCH agent slot 1    -> 可选子 agent 只读 ext4
host:  Resume
guest: mount /workspace
guest: mount /opt/agents/0, /opt/agents/1
host:  run turn

这里有两个硬规则。

第一，slot 身份稳定。比如 vdb 永远是 workspace，vdc、vdd、vde 是 agent slots。设备语义不能在不同启动里漂移，否则 guest-agent 的挂载和排障都会变复杂。

第二，mount 延迟到 resume 后由 guest-agent 做。golden 里只保留设备存在性，不提前把占位盘挂成 /workspace 或 /opt/agents/N。Firecracker 允许换 backing file，不代表 guest 文件系统缓存会忘掉旧盘；如果快照里已经有挂载态，resume 后再换盘，就是让 guest 带着旧文件系统状态看一块新盘。

这套机制的实测成本很小：

操作 实测
snapshot/load 约 16ms
每块盘 PATCH 换 backing 约 9ms
guest 串行 mount 两个只读卷 约 6ms
多卷净增 约 15-20ms

它落在约 443ms 的取用预算里，换来的是一份 agent-agnostic 的 golden snapshot：agent 升级不重烤 rootfs，同一个 VM 可以挂主 agent 和子 agent，只读 agent 卷可以跨 VM 共享 page cache。

软件不要和 agent 抢设备槽

agent 走只读卷，不代表所有能力都应该走卷。Firecracker 的设备槽有限，软件、runtime、工具链又是长尾：版本多、用户选择多、安装方式也多。如果每个软件版本都变成一块 VM 设备，槽位会先耗尽，平台还要维护大量镜像。

我把 agent 和软件分开处理。agent 是平台受控发布物，数量少，需要版本化和回滚，适合只读 virtio-blk 卷。软件、runtime、工具链作为 workspace 文件写入 /workspace，随 workspace 一起持久化，通过 PATH 等环境变量启用。这个方案没有“所有东西都模块化挂载”整齐，但它尊重 Firecracker 的设备模型：块设备留给少数边界清楚的发布物，长尾状态进入 workspace。

持久化也不能破凭证边界

workspace 最终要落到对象存储，但 S3 凭证不能进 VM。持久化必须在 host 侧完成。

运行时，VM 内的 /workspace 是本地 ext4。host 对 workspace 块设备做 dm-snapshot，拿到只读视图，再把这个视图挂到 host 上，用 host 自己的云凭证执行 aws s3 sync，把文件级增量同步到对象存储。生命周期分三段：

启动前：对象存储原始文件树 -> host 本地运行目录 -> workspace ext4
运行中：VM 写 /workspace -> host dm-snapshot -> aws s3 sync
停机/checkpoint：host 收口同步 -> 可选 fsfreeze -> 对象存储成为恢复真源

S3 存原始文件，不存整块 ext4 镜像。ext4 是运行时载体，原始文件树是持久化真源。这样用户离开平台时拿到的是目录树和文件名，不是只能由平台解释的整盘镜像。

这条路径有几个实现细节决定它能不能可靠运行。

第一，host 不能直接挂 live ext4。那块 ext4 正被 Firecracker 作为 guest block device 使用，host 并发挂载会读到脏 journal 或陈旧元数据。dm-snapshot 给 host 一个短命 COW 视图：guest 继续写 origin，host 只读 snapshot，两边不互相踩。

第二，高频同步不做 fsfreeze。实测 fsfreeze 冻结窗口约 260ms；每 5s freeze 一次，对 npm install、build、大量解压写入会形成周期性顿挫。后来我把一致性拆成两层：高频持久化默认做不 freeze 的 dm-snapshot，拿类似突然断电后的崩溃一致视图；turn 结束或用户显式保存版本时，再做 fsfreeze + dm-snapshot，拿更强的静默一致性。

第三，workspace 块设备需要 cache_type: Writeback。默认配置下，即使 guest 静止并执行 fsfreeze，host 快照仍可能落后 guest 已经 flush 的数据。改成 Writeback 后，guest flush 能落到 host fd 的 fsync，host 侧快照和 guest 视图才能对齐。

第四，运行中同步对删除要保守。高频 sync 尽快推新文件和改动，但不要因为一次中间态扫描就删除远端对象；停机、checkpoint 或明确收口点，再做更强的删除同步。对用户数据来说，远端多留一个旧文件通常比误删更容易恢复。

这条链路和模型调用路径的原则一致：VM 负责产生文件系统状态，host 负责把状态转成持久化事实；VM 始终没有对象存储凭证。

VMM 进程本身也要隔离

microVM 不是唯一边界。Firecracker VMM 是 host 进程，生产里也要有自己的进程级隔离。

Firecracker design 推荐通过 jailer 启动 Firecracker。jailer 设置 chroot、cgroup，丢权限，再 exec Firecracker；Firecracker 自己也会安装 seccomp 过滤器，限制 host syscall。

我的 host 侧按这个模型组织：每个 microVM 有自己的 jailer chroot，rootfs、workspace 设备、agent 卷、vsock UDS 都放在这个边界里。egress identity sidecar 也跟 chroot 同生命周期：VM 启动时写入，VM 停掉后 chroot 清理，egress proxy 下一轮 reconcile 就撤掉对应 listener。

这样即使 VMM 自身出问题，外层还有 chroot、cgroup、seccomp 和权限下降兜住下一层。

最后留下来的原则

Firecracker 的价值和限制来自同一件事：它足够小。小设备模型带来更窄攻击面，也迫使系统围着边界设计。

安全这条线最后收束成一句话：VM 可以执行代码，但不能持有能力。模型调用由 host egress proxy 重签，S3 持久化由 host 做，VM 没有普通网卡，管理和出向都走 vsock。agent loop 可以进入 VM，因为真正的能力仍然在 VM 外面。

分发这条线也类似：agent 不进 rootfs，不依赖共享目录，而是按 agent × 版本 打成只读 virtio-blk 镜像。MMIO snapshot 路径不能 resume 后新增盘，就在 golden snapshot 里预留槽位，Paused 阶段 PATCH 换 backing。设备槽留给少量平台发布物，软件、runtime、工具链进入 workspace。

状态这条线则是把“当前态”和“持久化真源”拆开：VM 内 ext4 是在线当前态，host 用 dm-snapshot 提取文件级视图，再同步到对象存储；VM 始终没有对象存储凭证。Firecracker VMM 自己再用 jailer、chroot、cgroup、seccomp 关一层。

边界内的东西越少、越静态，系统越容易守住。把凭证和网络出口放在 VM 外，把动态设备需求前移到模板期，把长尾状态放进 workspace，Firecracker 才能同时服务启动速度和安全隔离这两个目标。

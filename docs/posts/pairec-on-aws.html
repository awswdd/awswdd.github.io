工程笔记

在 AWS 上落地一套生产级推荐系统:基于开源 PaiRec 的全链路改造实践

内容创作：wdd · 美化编辑：codex (gpt-5.5) · 审核：wdd

约 5527 字 · 阅读约 16 分钟

把一套 PAI-Rec 风格的推荐链路(召回 → 过滤 → 排序 → 重排 → 回流 + A/B 实验)完整搬到 AWS。 这篇文章不讲"推荐算法",讲的是工程落地:AWS 上做推荐缺什么、有哪些落地路线、为什么选 PaiRec 引擎、 落地时撞到哪些问题、具体怎么改造(含代码量与工作量评估)、最终跑出来什么样。

全部内容来自一套已在真实 AWS(ap-south-1)端到端跑通的实现,数字可溯源到代码与设计文档。

1. 在 AWS 上做推荐系统,缺的是什么

先说一个容易被忽略的事实:AWS 没有一个"推荐引擎"产品。

AWS 上和推荐沾边的托管能力,要么是端到端黑盒,要么是散装积木:

  • Amazon Personalize 是端到端托管推荐,你喂数据、它出结果。好处是快;代价是召回/排序逻辑是黑盒,无法定制多路召回的具体策略、无法插自己的重排规则、A/B 也得用它那套。对"已经有算法团队、要白盒可控"的场景不合适。
  • 其余都是积木:SageMaker(训练/推理)、OpenSearch(向量检索)、ElastiCache(KV)、Glue(数据处理)、MSK(事件流)、Athena(分析)。每块都很强,但它们之间没有"推荐编排层"。

而推荐系统真正的复杂度,恰恰就在这个编排层:

flowchart TD
  A["多路召回并发<br/>u2i · i2i · 向量 · 热门 · 兜底"] --> B["结果合并 + 去重 + 打散"]
  B --> C["按版本取特征"]
  C --> D["拼成模型输入 → 精排打分"]
  D --> E["重排<br/>业务置顶 · 多样性 · 强插"]
  AB["A/B 分流贯穿全程 + 埋点归因"] -.-> A
  AB -.-> D
  AB -.-> E

这套编排框架——多路召回怎么并发、候选怎么去重打散、特征按什么 key 取、打分协议怎么对接、重排规则怎么插拔、A/B 怎么正交分层——AWS 不提供,Personalize 把它藏进黑盒,而自己从零写要持续投入很久。

所以"在 AWS 上做推荐"缺的不是算力,而是这层编排引擎,以及把引擎和各托管件接起来的适配层。 这篇文章的主线,就是怎么补上这一层。

2. 落地 AWS 的三条路线

路线 做法 优点 代价
A. 全托管 Amazon Personalize 最快、零运维 黑盒,召回/排序/A/B 不可定制;迁移已有算法资产困难
B. 自建引擎 自己写编排引擎 + 接积木 完全可控 编排框架(16+ 召回 / 10+ 过滤 / 9+ 排序 / A/B)要长期投入维护
C. 开源引擎 + 托管积木 拿成熟开源编排引擎,只写"接 AWS 的适配层"+ 自己的算法 编排能力开箱即用、可白盒定制、团队只维护薄适配层 需要把开源引擎适配到 AWS(本文重点)

本方案走 路线 C:编排引擎用开源件,把精力投在"接 AWS"和"自己的算法"上。下面先说为什么是 PaiRec,再说适配怎么做。

3. 为什么选 PaiRec,以及它的工程优点

alibaba/pairec 是阿里云开源的推荐引擎(Go,Apache-2.0)。选它的工程理由:

3.1 编排能力开箱即用,覆盖面远超起步自研

PaiRec 内置了推荐编排的全套能力:十多种召回类型、十多种过滤器、近十种排序/重排策略,以及完整的 A/B 实验框架。这意味着"多路召回并发 → 去重打散 → 特征 → 精排 → 重排"这条主链,配置即可跑通,不用写编排代码。

3.2 配置即编排,且与 PAI-Rec 同源

PaiRec 用一份 JSON 描述"场景 → 召回链 → 过滤链 → 特征 → 排序链",这份配置格式就是阿里云 PAI-Rec 的编排格式。对从阿里云迁移的团队,配置几乎可平移;对新团队,改 JSON 就能调编排,不用动代码、不用重新编译。

// pairec.json 片段:一个场景的编排全靠配置描述
{
  "SceneName": "home_feed",
  "RecallNames": ["u2i_custom", "i2i", "vec_knn", "global_hot"],
  "FilterNames": ["exposure", "snake_filter"],
  "SortNames": ["algo_score_sort", "pin_sort", "category_diversity"]
}

3.3 提供"注册函数 + Hook"扩展点,不改源码就能扩展

这是落地 AWS 的关键。PaiRec 暴露了 AddStartHook + 一组 Register*(RegisterRecall / RegisterFilter / RegisterSort / RegistRequestDataFunc),允许你注册自己的算子实现而不碰引擎源码。我们所有的 AWS 适配(Redis 召回、OpenSearch 向量、SageMaker 打分适配、HTTP 算法代理),都是通过这个扩展点挂进去的——PaiRec 源码零修改。

3.4 Go 实现,容器化友好

镜像小、启动快、内存占用低,适合 EKS 上多副本横向扩展。

小结:PaiRec 把"编排"这块最重的活儿用成熟开源件解决了,还留了干净的扩展口子。我们要做的,是给它接上 AWS 的数据源和推理端,并补上 A/B 分流与离线数据链路。

4. 落地 AWS 时撞到的真实问题

把 PaiRec 接到 AWS,有一串"开源默认形态"和"AWS 托管现实"之间的鸿沟需要填:

# 问题 本质
P1 PaiRec 召回/特征默认对接阿里云 Hologres/FeatureStore 数据源要换成 ElastiCache Redis / OpenSearch
P2 PaiRec 精排默认发 TFServing 协议、不做 SigV4 签名 SageMaker Endpoint 强制 SigV4,协议和鉴权都对不上
P3 PaiRec 内置 A/B(PaiABTest)是阿里云专有 A/B 分流、正交分层、归因要自建
P4 算法工程师用 Python/Java,PaiRec 是 Go 算法无法直接嵌入引擎,需要语言无关的接入方式
P5 训练样本特征(离线 Spark)与在线推理特征(在线服务)两套实现 训练-服务偏差:线上效果悄悄打折且难排查
P6 离线数据处理早期"Spark 塞进 SageMaker Processing 容器" 不是大数据生产形态:无数据目录/质量/调度,版本易碎
P7 A/B 配置裸存 Redis,改了立刻生效 控制面无版本、不可审计、不能回滚;在线与端侧可能配置漂移
P8 推荐不只在云端,端侧(手机)也要做轻量召回/粗排 配置/特征/向量/模型要安全下发到不可控的海量设备
P9 团队没有专职 K8s 运维,但请求量会增长 部署要免运维且能弹性扩展

接下来逐个说怎么改。

5. 适配与改造:具体方案、代码量、工作量评估

整套改造遵循一条原则:PaiRec 本体不改,只在外围加适配层和自己的服务。下面按模块拆开,每节给出方案、关键代码、真实代码量,最后汇总工作量评估。

代码量为本仓库实测行数(含注释与配置;Go/Java/Python 混合)。工作量为按模块复杂度的经验估算(人天),包含开发 + AWS 资源搭建 + 联调,供规划参考。

5.1 引擎适配层:Redis 召回 + OpenSearch 向量 + 不改源码扩展(应对 P1、P3 部分)

PaiRec 的 main.go 实际只有两句"引擎调用"——pairec.AddStartHook(...) 注册扩展,pairec.Run() 启动。所有 AWS 适配都在 hook 里:

pairec.AddStartHook(func() error {
    algorithm.RegistRequestDataFunc("rank_model", tfRequestDataFunc)  // 精排数据适配(见 5.2)
    recall.RegisterRecall("country_v0", NewRedisGroupHotRecall(conf)) // Redis 分组热门召回
    filter.RegisterFilter("exposure", NewExposureFilter(expConf))     // 曝光去重(读 Redis 已曝光集合)
    sort.RegisterSort("pin_sort", NewPinSort(...))                    // 运营置顶(读 rec:pin:<scene>)
    return nil
})
pairec.Run()

配置侧用"合法占位类型 + 标记字段"通过引擎校验,hook 启动时识别标记并用自写实现覆盖注册——这样既复用引擎的场景/校验机制,又替换了底层数据访问。

向量召回单独成一个文件:从用户向量发起 OpenSearch k-NN 查询,走 VPC 内网 + SigV4。

涉及文件 / 代码量: - pairec-poc/engine/main.go(扩展注册 + 配置加载):317 行 - pairec-poc/engine/http_operators.go(HTTP 代理算子,见 5.4):258 行 - pairec-poc/engine/opensearch_recall.go(OpenSearch k-NN 召回):320 行 - 引擎扩展合计:约 895 行 Go

工作量评估:3–5 人天(熟悉 PaiRec 扩展接口后,主要成本在 OpenSearch SigV4 + Redis DAO 调试)。

5.2 SageMaker 精排适配:SigV4 sidecar 代理(应对 P2)

PaiRec 发 TFServing 风格的 JSON,不带签名;SageMaker Endpoint 要求 SigV4。直接改 PaiRec 的 HTTP 客户端会侵入源码,我们用同 Pod 的 sidecar 代理解决:

flowchart LR
  P["PaiRec"] -->|"普通 HTTP · localhost:9000"| S
  subgraph S ["smproxy（sidecar）"]
    direction TB
    S1["TFServing JSON → SageMaker 调用"] --> S2["SigV4 签名"]
  end
  S -->|"已签名请求"| E["SageMaker Endpoint<br/>rec-ranker"]
  E -.->|"scores[] 回填 item.score"| P

引擎侧只需把精排地址指向 localhost:9000,完全不知道下游是 SageMaker。RegistRequestDataFunc("rank_model", ...) 负责把特征按固定顺序拼成模型输入张量。

涉及文件 / 代码量: - pairec-poc/proxy/proxy.py(SigV4 + 协议转换):75 行 Python - 引擎侧 tfRequestDataFunc(在 main.go 内):约 40 行

工作量评估:1–2 人天(SigV4 + TFServing↔SageMaker 协议对齐)。

5.3 A/B 分流服务 ab-router:正交分层 + scene 翻译 + 埋点(应对 P3)

PaiRec 内置 A/B 是阿里云专有,我们前置一个独立的分流服务 ab-router(Java/Spring Boot),把 A/B 的复杂度全收在这里,让 PaiRec 退化为纯执行引擎:

一次请求在 ab-router 做三件事: 1. 正交分层分桶:对每个层(召回层 / 排序层)独立分桶: bucket = murmur3_32(uid + "#" + scene#layer) % 1000(salt = 场景#层名,基数 1000 = 最细 0.1% 流量切;还支持互斥域 Domain 做场景隔离)。同一用户在每层稳定落桶,各层正交——可同时参加多个层的实验互不污染。 2. scene 翻译:把"命中哪些实验"翻译成 PaiRec 的 scene_id(如 recall=grouphot & rank=v2 → home_feed_v2),再调引擎。 3. 埋点:曝光事件带 request_id / exp_id / layer_hits 发 MSK,fire-and-forget 不阻断主路。

exp_id 串成 home_feed#recall:exp_grouphot#rank:exp_v2 随曝光回流,供离线按层归因。

涉及文件 / 代码量(只统计 A/B 核心链路;ab-router 整个模块约 4676 行,其中 ~3200 行是演示控制台的 6 角色工作台后端,不计入核心链路): - abtest/ 包(ABRouter 分桶 356 + Domain/Experiment/Layer/ExpResult):约 573 行 - web/RouterController.java(请求编排 + 埋点):142 行 - web/PaiRecClient.java(scene 翻译 + 调引擎):106 行 - A/B 核心合计:约 821 行 Java

ab-router 模块里另有两块也计入总账,但归到各自小节:ControlPlaneStore(控制面,见 5.9)、EdgeDistController(端侧分发,见 5.10);三块相加 821+214+430=1465 行,即 ab-router 进核心链路总账的部分。

工作量评估:6–8 人天(分桶算法与离线/端侧逐位对齐、MSK IAM 鉴权埋点、scene 翻译表是主要成本)。

5.4 自定义算法服务化:HTTP 代理算子 + 独立算法服务(应对 P4)

算法工程师不该被 Go 绑住。我们在 PaiRec 侧加一个通用 HTTP 代理算子(召回/过滤/排序各一个),按固定契约调外部服务;算法本身写成独立服务,语言任意:

flowchart LR
  P["PaiRec（Go）"] -->|"HTTP POST /recall<br/>{uid, scene, size, features}"| R["recall-svc<br/>（本例 Java / Spring Boot）"]
  R -.->|"{items:[{id, score, retrieve_id}]}"| P

接入只需三步:① 算法服务打镜像部署成 EKS Service;② 引擎设 ALGO_SVC_URL 环境变量(hook 自动注册 http_recall/http_filter/http_sort);③ pairec.json 里把某路召回写成 http_recall。算法迭代 = 重打算法服务镜像,引擎不动;换 Python/C++ 也行,遵循契约即可。

涉及文件 / 代码量: - pairec-poc/engine/http_operators.go(通用代理,已含在 5.1 的 895 行):258 行 - recall-service/(Java 算法服务模板):151 行 Java

工作量评估:2–3 人天(代理算子 + 契约 + 模板服务;之后每个新算法服务按业务复杂度另算)。

5.5 端云一致:特征算子封装成 Spark UDF(应对 P5)

训练样本特征(离线)和在线推理特征(在线)如果是两套实现,就会有训练-服务偏差——同一个 item 训练看到的 quality_z 和线上不一样,模型上线不涨还查不出原因。

解法:特征代码只写一份,两端共用。把特征生成逻辑封装成 feature-ops 模块(一个 FeatureGenerator + 一份 feature_config.json):

  • 在线:online-service 直接调 feature-ops 算特征;
  • 离线:把同一个 jar 包成 Spark UDF(feature_gen),在 Glue 的样本生成 SQL 里调:
SELECT feature_gen(request_time, price, quality, ctr) AS f, label
FROM labeled JOIN item_wide USING(item_id) JOIN user_wide USING(user_id)
-- f[0..4] = day_h, week_day, price_bucket, quality_z, ctr_z(顺序与在线模型严格对齐)

这样"数据处理用 SQL,特征用同一份 jar",可读性和一致性兼得。

涉及文件 / 代码量: - feature-ops/(共用特征算子):300 行 Java - offline-spark/FeatureGenUDF.java(把算子包成 Spark UDF):86 行 Java

工作量评估:3–4 人天(关键在 UDF 适配 + 验证两端逐字节一致)。

5.6 离线数据处理迁 Glue + Glue Workflows(应对 P6)

早期数据处理是 Spark 跑在 SageMaker Processing 容器里——能跑,但要自己打 fat jar、版本易碎、调度靠外挂、没有数据目录/质量这些数据工程能力。按生产标准迁到 Glue + Glue Workflows(serverless Spark,最贴 DataWorks 体感):

  • 算力:Glue serverless Spark,免管集群;
  • 编排 + 定时:Glue Workflow 自带 cron + 依赖触发 + 可视化 DAG(无需外挂 Scheduler);
  • 数据目录:Glue Crawler 编目 → Glue Data Catalog(与 Athena 共元数据);
  • 数据质量:Glue Data Quality 用 DQDL 规则集做门禁。

真实 DAG(Workflow rec-offline-glue-workflow,每天 02:00 UTC):

  B --> S["样本生成<br/>label(SQL) + 特征(feature-ops UDF)"]
  B --> I["i2i<br/>ItemCF 共现 + 余弦"]
  B --> V["向量<br/>Spark ML ALS"]
  S --> Q["质量门禁<br/>Glue Data Quality 规则集"]
  I --> Q
  V --> Q
  Q -->|"不达标"| X["作业失败<br/>不触发训练"]

数据处理主体用 Spark SQL 表达(可读);i2i 用 ItemCF 共现+余弦(top-50),向量用 Spark ML 的 ALS(隐式反馈矩阵分解,rank=16,比单机 SVD 可扩展)。

涉及文件 / 代码量: - pipeline/glue/(5 个 Glue 作业:数据加工 106 / 样本 86 / i2i 78 / 向量 72 / 质量门禁 92):434 行 Python - offline-spark/(DataProcessing + SampleGeneration,旧 Spark 逻辑可复用为参考):409 行(含 5.5 的 UDF)

工作量评估:6–9 人天(5 个作业改写为 Spark SQL + Glue Data Quality 规则 + Workflow 触发器编排 + 端云一致 UDF 联调)。

5.7 数据段→模型段衔接 + SageMaker 只做训练注册(应对 P6 延伸)

数据处理迁 Glue 后,SageMaker Pipeline 只保留训练 + 注册两步,输入直接吃 Glue 产出的样本:

flowchart TD
  G["Glue 质量门禁 SUCCEEDED"] -->|"EventBridge 规则"| P["StartPipelineExecution<br/>rec-offline-pipeline"]
  P --> T["TrainRanker<br/>SKLearn · 读 s3://.../samples"]
  T --> R["RegisterModel → Model Registry<br/>rec-ranker-group · 审批 Approved 才部署"]

EventBridge 规则匹配 Glue Job State Change = SUCCEEDED(质量门禁作业)→ 触发训练。两段松耦合、各自可独立重跑,质量不达标连训练都不会启动。

涉及文件 / 代码量: - pipeline/build_pipeline.py(训练+注册 Pipeline 定义):98 行 - training/train.py + inference.py:132 行 - EventBridge 规则:配置(非代码)

工作量评估:2–3 人天。

5.8 数据面真相源:S3 源 + 幂等上线作业(应对 P6 延伸)

一条生产原则:在线存储(Redis/OpenSearch)不是数据的家,只是可重建副本。

flowchart LR
  G["Glue 产出"] --> S["S3<br/>Parquet 源 · 持久真相"]
  S --> Y["seed_sync<br/>SageMaker Processing · 幂等"]
  Y --> R["Redis / OpenSearch<br/>可重建副本"]

特征/向量/召回先落 s3://<data>/seed/ 的列式 Parquet;上线作业读 S3 写 rec:feat:item:vN:* / rec:u2i:* / rec:grouphot:*(Redis)和 item-vectors k-NN 索引(OpenSearch)。Redis 挂了、索引坏了,重跑一次就从 S3 恢复,不丢数据;S3 路径 seed/item_features/vN/ + Redis key 前缀 rec:feat:item:vN: 天然支持特征版本 A/B。

涉及文件 / 代码量: - pipeline/featuresync_src/seed_sync.py(幂等上线作业):186 行 Python

工作量评估:2–3 人天。

5.9 控制面 RDS 统一发布(应对 P7)

A/B 配置这类控制面数据裸存 Redis 不安全(无版本、不可回滚)。迁到 RDS PostgreSQL + 统一发布流程:

三张表——ab_config_draft(草稿,可改不生效)、published_version(已发布版本,不可变,带 SHA256)、published_pointer(当前版本指针)。改配置 = 改草稿 → 发布成版本 → 在线 ab-router 与端侧手机都消费"当前已发布版本";回滚 = 切指针(秒级),在线和端侧同时回退。Redis 退化为已发布版本的热缓存,ab-router 每 30s 或发布后立即从 RDS 重读。

涉及文件 / 代码量: - ab-router/.../control/ControlPlaneStore.java(三表 + 发布/回滚/版本):214 行 Java

工作量评估:3–4 人天。

5.10 端侧分发:S3 + CloudFront 签名 URL + 增量包(应对 P8)

端侧(手机)要做轻量召回/粗排,需安全下发四类制品:A/B 配置、特征、向量、端侧模型。面向千万级不可控设备:

  • 存储:制品版本化存 S3(edge/<kind>/<kind>-vN.json),元数据(版本/SHA256/大小)进 RDS;
  • 分发:CloudFront(边缘缓存抗大基数)+ 签名 URL(TrustedKeyGroup,绑路径 + 10 分钟过期);
  • 增量:特征/向量产增量包(相对上一版的 changed/removed),手机差一版时只下增量,省带宽;
  • 校验:每包带 SHA256,端侧重算比对,不过用本地旧版(绝不开天窗),下临时文件 → 校验过 → 原子 rename;
  • 端云分桶一致:端侧 SDK 复刻同一份 murmur3_32 % 1000 + 同 salt,实测 u0000123 端云都落桶 187、u9999999 都落 379。

防盗链实测(三层防护):S3 完全私有(仅 CloudFront OAC 可读)+ 强制签名 URL + 10 分钟过期。攻击实测:直连 S3→403;无签名→403 MissingKey(签名校验在缓存命中之前);篡改路径/Expires/Signature→全 403;合法签名 10 分钟内→200。

涉及文件 / 代码量: - ab-router/.../web/EdgeDistController.java(发布/清单/增量/签名/回滚/分桶):430 行 Java

工作量评估:5–7 人天(CloudFront OAC + 签名密钥 + 增量包 diff + 端云分桶对齐 + 防盗链验证)。

5.11 部署:EKS Auto Mode + TargetGroupBinding(应对 P9)

在线跑在 EKS Auto Mode(集群 rec-eks / 命名空间 rec)。选它的理由很实在:团队没有专职 K8s 运维,而请求量会增长——Auto Mode 全自动管节点(扩缩/打补丁/替换/bin-packing)+ 托管周边组件。用户只需两步:① 控制台「镜像构建」触发 CodeBuild 推 ECR;② 「EKS 发布」选服务填镜像 tag 滚动更新。其余(调度/节点/扩缩/服务发现/健康检查)全交给 Auto Mode。

两个部署细节: - 服务发现回归 K8s 内置 DNS(pairec.rec.svc.cluster.local),不需 Cloud Map; - 入口稳定用 TargetGroupBinding 而非 Ingress:ALB 由 AWS 控制台独立管(外网 IP 不变、白名单不失效),controller 只把 ab-router 的 Pod IP 同步进现有 target group。

涉及内容 / 工作量:eksctl 集群配置 + K8s manifests + TargetGroupBinding + IRSA,约 4–6 人天(主要在 IRSA 权限调试与 ALB 解耦)。

6.3 测试效果

测试分两方面:功能——支持哪些推荐系统能力(均为真实 AWS 实测);性能——在线链路的实测延时与吞吐。

6.3.1 功能:支持的推荐系统能力(真实实测)

召回(多路并行)——以下召回类型均实测命中对应数据:

召回 实现 数据源 实测
u2i 个性化 UserCustomRecall Redis rec:u2i:<uid> 返回该用户历史相关 item
i2i 协同过滤 ItemCF(离线 Glue 算) Redis rec:i2i: 按最后点击 item 出簇内同伴
向量召回 OpenSearch k-NN OpenSearch 向量索引 同兴趣簇内近邻(SigV4 VPC 内网)
分组热门 自写 Redis 召回 rec:grouphot:country:<C> 用户特征(国家)驱动分组 key
全局热门兜底 无 Trigger 召回 rec:globalhot:all 冷启动用户(无 u2i)正常出结果

过滤 / 排序 / 重排——均实测生效: - 过滤:曝光去重(读 Redis 已曝光集合,已曝光 item 被滤掉)、SnakeFilter 多路蛇形打散; - 精排:特征 → smproxy SigV4 → SageMaker Endpoint 打分,结果按模型分重排(打分最高的 item 冲到第 1); - 重排:运营置顶(PinSort 把垫底 item 提到最前)、多样性打散(同类目交错); - 在线派生特征:day_h(请求时实时算的小时)等上下文特征在请求时现算。

特征版本化 A/B:home_feed(特征 v1)vs home_feed_v2(特征 v2)对同一用户产出不同排序——验证特征按版本隔离、A/B 可切版本。

A/B 正交分层 + 归因闭环(合成数据注入已知效应,验证归因链路能检出): - 召回层、排序层正交分桶,exp_id 形如 home_feed#recall:exp_grouphot#rank:exp_v2; - 埋点 expose/click/convert 经 MSK→Firehose→S3→Athena 落库,按 layer/variant 聚合 + 双比例 z-test; - 实测:分组热门组 CTR 显著高于对照组(z≈2.12,95% 显著),CVR 同向(16.7% vs 9.1%)——分流 → 归因 → 显著性的飞轮闭环可用。

离线 + 飞轮: - Glue Workflow rec-offline-glue-workflow 6 个动作全部 SUCCEEDED(编目→数据加工→样本/i2i/向量并行→质量门禁); - 端云一致 UDF 在 Glue 中验证产出特征列 day_h/week_day/price_bucket/quality_z/ctr_z/label; - 质量门禁通过 → EventBridge 触发 → SageMaker Pipeline 新执行 Succeeded。

容错降级:注入 Redis 故障 → 自动走全局热门兜底,不开天窗;端侧下载校验失败 → 用本地旧版。

6.3.2 性能:在线链路实测延时与吞吐

用内置压测端点 /admin/demo/benchmark 在 EKS 上对全链路并发施压(ab-router 进程内 → 分流 → PaiRec → Redis 多路召回 + OpenSearch + SageMaker 精排,含各数据源真实网络往返;测的是服务端处理,不含公网 RTT),实测:

并发 请求数 成功率 QPS P50 P90 P99 Max
20 2000 100% 360 29ms 167ms 237ms 553ms
50 3000 100% 579 49ms 212ms 240ms 346ms
  • 成功率 100%,无失败请求;
  • 并发从 20 提到 50,QPS 从 360 升到 579(单 Pod、未横向扩),P99 稳定在 ~240ms;
  • 这是单副本的数字——EKS Auto Mode 下横向加副本即可线性提吞吐。

分阶段耗时与瓶颈定位(典型量级,用于找优化重点):

阶段 组件 P50 P99
分流 ab-router 一致性哈希 1ms 2ms
召回 Redis 多路 + OpenSearch k-NN 8ms 18ms
过滤 曝光去重 / SnakeFilter 2ms 5ms
特征 Redis 批量取 + 在线派生 6ms 14ms
精排 SageMaker Endpoint 打分 28ms 95ms
重排 多样性打散 / 运营置顶 3ms 8ms

瓶颈是精排(SageMaker 远程打分,占整链路约 60% 耗时)——这也符合推荐系统的普遍规律:延时大头在模型计算与特征/打分的网络往返,与编排框架本身无关。对应的优化方向很清晰:模型蒸馏 / 批处理打分 / 打分结果缓存 / 把轻量模型进程内推理(省一次网络往返)。

测试条件说明:压测在 ap-south-1 真实 EKS 环境、单 ab-router 副本、home_feed 场景、随机 uid(覆盖冷热用户)下进行;数据为注入已知 A/B 效应的合成数据,目的是验证工程链路的吞吐/延时/正确性/可归因性,而非真实业务增益。接真实流量后,同一套链路与压测方法即可产出生产基线。

端侧: - 四类制品经 CloudFront 签名 URL 下载 HTTP 200 + SHA256 校验一致;增量包正确合并; - 防盗链:无签名/篡改路径/篡改 Expires/篡改 Signature 全部 403; - 端云分桶逐位一致(187/379 实测对齐)。

说明:测试用的是结构化合成数据(注入了已知 A/B 效应),目的是验证工程链路的正确性与可归因性,不是真实业务增益;接真实流量后,同一套链路即可产出真实指标。

7. 几个可复用的工程判断

  1. AWS 上做推荐,先补"编排层"——算力是积木,编排才是缺口;用成熟开源引擎 + 扩展点接 AWS,比从零造或用黑盒都更可控。
  2. 适配层别侵入引擎——AddStartHook + Register* + sidecar 代理,让引擎源码零改动,升级开源版本无痛。
  3. 算法服务化、语言无关——引擎只编排,算法独立成 HTTP 服务,谁的语言都能接、能独立扩缩热更。
  4. 端云一致用同一份特征代码——离线把在线的特征 jar 包成 Spark UDF,从根上消除训练-服务偏差。
  5. 数据真相在对象存储,在线存储是可重建副本——挂了能重跑恢复。
  6. 控制面版本化 + 统一发布——草稿/版本/指针,在线与端侧同源、秒级回滚。
  7. 数据用 Glue、模型用 SageMaker、EventBridge 衔接——各用所长,数据段自带 cron 与质量门禁。
  8. 没有运维就上 EKS Auto Mode + TargetGroupBinding——用户只管镜像版本,入口与节点都交给托管。
